# twin-ship — the house twin, packaged and run as a stranger's artifact

Roadmap Nice-to-Have #7 (the twin-ship packaging skill). The base `godot-export-builds` exports the
pck/binary only — game-shaped, no concept of runtime data files. `twin_ship.sh` adds the twin's
deploy contract: the export-safe viewer (phase 1) PLUS its swappable data tree (`model` + sidecar +
binding map + recordings) staged **beside** the build, so a site retargets `url=`/`model=` **without
re-exporting**. This is the Phase-3 clean-stranger validation of that tool + skill from the twin seat.

## Evidence provenance (what ran where)

Run from the **twin seat** (`twindemo/house/`), NOT the framework repo. The seat's house viewer was
temporarily overlaid with the branch's export-safe viewer files (`main.gd`, `core/data_bus.gd`,
`project.godot` from `feat/twin-ship`) and the branch's `tools/twin_ship.sh`, plus a hand-authored
`export_presets.cfg` (the recommended macOS + a Linux preset). Every overlaid file was md5-backed up
and byte-restored afterward; the recommended preset, the tool copy, and all `dist/` output were
removed — the seat was left exactly as-found (`git status` unchanged). The **artifact** (the zip) was
unzipped to a clean scratch path **outside any project** and run there, following only the shipped
`README.txt` — the clean-stranger rule applied to the deliverable. Machine: Apple M3 Pro, macOS Darwin
25.5, Godot **4.6.3.stable** (`/Applications/Godot.app`), export templates 4.6.3.stable installed.

## Artifact sizes (real, this run — house twin over Duplex_A)

| piece                  | size                     | note                                                                                                                      |
| ---------------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| `duplex.app` (total)   | 178 MB (186 355 712 B)   | dominated by the executable                                                                                               |
| in-bundle executable   | 176.2 MB                 | the **universal** engine template binary (x86_64 + arm64 slices) — NOT your data                                          |
| `.pck` (code + scenes) | **75 380 B (73.6 KB)**   | code + starter scenes ONLY — data excluded, the contract held                                                             |
| `data/` (beside build) | 1.4 MB                   | `duplex.glb` 484.3 KB + `duplex_props.json` 642.7 KB + `binding_map.json` 3.5 KB + `recordings/house-day.ndjson` 261.8 KB |
| `duplex-macos.zip`     | **60 MB (61 865 529 B)** | deterministic (fixed sort order + `zip -X`); the 176 MB binary compresses ~2.9×                                           |
| Linux `duplex.x86_64`  | 68 MB (71 075 864 B)     | flat layout (`viewer.cfg` + `data/` + `README.txt` siblings, no bundle)                                                   |

The pck is **75 380 bytes** — code + starter scenes only. (The plan's carry-forward cited a 46.9 KB pck
from the phase-2 minimal viewer; the house viewer ships more code — `overlay/`, the timeline scrub bar,
five `core/` scripts — so 75 KB is the honest house number. The point stands: the ~1.4 MB of data is
NOT in the pck; it sits in `data/` beside the build.) The macOS ship (preflight → export → assemble →
smoke → zip) took **~10.6 s wall** (`time`: 10.583 s total, 6.55 s user).

## The layout contract (data beside the build, inside the bundle on macOS)

The exported binary lives at `duplex.app/Contents/MacOS/`, so `viewer.cfg` and `data/` are staged
**there**, beside it — where the export-safe viewer roots them (`OS.get_executable_path().get_base_dir()`
in template builds). Assembled tree:

```
duplex-macos/
  README.txt
  duplex.app/Contents/MacOS/Xenodot Twin Viewer     # the executable (named by config/name, not the .app)
  duplex.app/Contents/MacOS/viewer.cfg              # rewritten to data/-relative paths
  duplex.app/Contents/MacOS/data/duplex.glb
  duplex.app/Contents/MacOS/data/duplex_props.json
  duplex.app/Contents/MacOS/data/binding_map.json
  duplex.app/Contents/MacOS/data/recordings/house-day.ndjson
  duplex.app/Contents/Resources/Xenodot Twin Viewer.pck   # code + starter scenes only
```

The assemble stage rewrote the shipped `viewer.cfg` `[viewer] model=` / `[twin] binding_map=` /
`recording=` to their `data/`-relative paths (ConfigFile-validate then line-rewrite, comments
preserved — the `twin_build.sh --wire` idiom), leaving `url=` untouched for a site to edit.

## The recommended preset — the two facts that matter

- **Universal arch, not arm64.** Godot 4.6.3 ships **no arm64-only macOS template binary** — a preset
  with `binary_format/architecture="arm64"` fails export ("template binary not found"). `universal`
  (x86_64 + arm64) is the safe default; it is why the executable is ~176 MB (it carries both slices),
  but that is the engine binary, not the twin's data.
- **`exclude_filter` keeps data OUT of the pck.** With
  `exclude_filter="*.glb,*.gltf,*.import,models/*,recordings/*,dist/*,build/*,tools/*,viewer.cfg,binding_map.json,*_props.json"`
  the pck is code + starter scenes only (75 380 B here). Without it the exporter sweeps `models/`,
  `recordings/`, the sidecar and the map into the pck and the data-beside-build contract is broken.
- **macOS ETC2/ASTC.** `textures/vram_compression/import_etc2_astc=true` is required under
  `[rendering]` in `project.godot` (macOS templates are ASTC-only; off, the export aborts). The
  export-safe viewer sets it (harmless in dev — runtime-loaded models are never texture-imported);
  `twin_ship.sh` asserts it in preflight rather than editing `project.godot` silently.

## Clean-stranger run — from the artifact, README instructions ONLY

Unzipped `duplex-macos.zip` to a clean path outside any project, then ran the README's direct-binary
command **verbatim**. Headless smoke log:

```
viewer: model loaded from data/duplex.glb
viewer: bindings resolved 6/6
viewer: playback of data/recordings/house-day.ndjson (59900 ms)
viewer: quit-after 20 frames reached
```

**One windowed boot** (no `--headless`, the README's windowed path) painted the twin on a real GPU:

```
Metal 4.0 - Forward+ - Using Device #0: Apple - Apple M3 Pro (Apple9)
viewer: model loaded from data/duplex.glb
viewer: bindings resolved 6/6
viewer: playback of data/recordings/house-day.ndjson (59900 ms)
```

Observed painting (viewer's own `--screenshot`, 1280×720): the Duplex house rendered — teal roof, grey
walls, green sensor markers — with the HUD reading `DataBus: PLAYBACK (recording)`,
`tags: 6 | entrance_door.open=0.189 seq=3`, `bindings: 6/6 resolved`, and the timeline scrub bar at
`0:00 / 0:59`. A painted, live-data-bound boot of the shipped artifact from `data/` beside the build —
the human-boot equivalent of acceptance 2.

## The data-swap contract, proven on the artifact (acceptance 3)

Editing the unzipped `viewer.cfg` `[viewer] model=` and re-running the **same** binary (no re-export):

| edit                                   | boot result                                                                            |
| -------------------------------------- | -------------------------------------------------------------------------------------- |
| `model="data/duplex.glb"` (baseline)   | `viewer: model loaded from data/duplex.glb`                                            |
| `model="data/NOPE.glb"` (neg. control) | `ERROR: viewer: failed to read model 'data/NOPE.glb' (error 7)` — loud, no silent pass |
| `model="data/duplex_renamed.glb"`      | `viewer: model loaded from data/duplex_renamed.glb`                                    |

The binary's mtime was unchanged across all three — the config is read fresh at runtime, not baked at
export. A site retargets `url=`/`model=` with a text edit; re-export only when the _code_ changes.

## Cross-platform SKIP (acceptance 4) — Linux export on a macOS host

A Linux preset exported and assembled (flat layout), and the smoke SKIPed **loudly**, exit 0:

```
twin-ship: PASS export (.../duplex.x86_64)
twin-ship: PASS assemble (dist/duplex-linux)
twin-ship: SKIP smoke — Linux build cannot run on this Darwin host (a SKIP is
  NOT a pass). Boot it on a Linux machine, or re-run there. To force anyway: --smoke.
```

A cross-platform build assembles and zips, but the smoke never fakes a green for a binary the host
can't run — `--smoke` would force it (and fail on the non-host ELF); the default SKIP is honest.

## Unsigned build — Gatekeeper stated honestly

The build is **not** code-signed or notarized (out of scope, `codesign/codesign=0`). On a Mac other
than the build machine, Gatekeeper warns on first open. The shipped `README.txt` says so plainly and
gives the two ways past it (right-click → Open → Open, or `xattr -dr com.apple.quarantine
duplex.app`). This finding does not claim the artifact is signed — it isn't.

## The clean-stranger test caught a real bug

The first artifact's `README.txt` printed the direct-run command as
`./duplex.app/Contents/MacOS/duplex` — the **model stem**. But the in-bundle executable is named by
`project.godot` `application/config/name` = `Xenodot Twin Viewer`, so a stranger pasting that line got
`no such file or directory`. (Phase 2 had fixed the _smoke_ to use the discovered binary name, but the
README still hardcoded the stem.) Fixed in `twin_ship.sh`: the README now prints the single-quoted full
path `'./duplex.app/Contents/MacOS/Xenodot Twin Viewer'`, which pastes verbatim. Re-shipped and
re-ran the clean-stranger test — the corrected command works. This is exactly what running the
deliverable as a stranger is for.

## Caveats

- **One machine** (Apple M3 Pro, Metal/Forward+, macOS Darwin 25.5, Godot 4.6.3). The universal
  binary's ~176 MB and the 60 MB zip are this engine version's template sizes; a single-arch preset
  would roughly halve the binary but is not the recommended default (arm64-only has no 4.6.3 template).
- **Sizes are the house twin over Duplex_A** — the pck (75 KB) scales with viewer code; the `data/`
  (1.4 MB) scales with the model + recording. A different twin ships different numbers; the _contract_
  (data beside the build, not in the pck) is what generalizes.
- **Linux/Windows smoke unproven on this host** — only the macOS smoke ran green (host-matched); the
  Linux artifact assembled but its boot must be verified on a Linux machine (the SKIP says so).
